## Part 2 — Architecture review

### The layers, as-is
1. **Tailscale mesh** — the de-facto fabric today. MagicDNS names are stable; IPs churn (gx10 `…0.97→0.59`, frogstation moved on reimage). *Dev/admin only* — not shippable to end users.
2. **Ansible management (item 0)** — version-controlled fleet ops. gx10 fully managed (SSH); frogstation pending the login fix, then OpenSSH-via-MSI to unify on SSH.
3. **Cloud coordinator (LokNet Router) — designed, not deployed.** Public Cloud Run front door; agents dial *out* (no inbound/NAT/mesh). This is what gives the *product* the "phone anywhere → home node" path **without** requiring users to run Tailscale, and gives each node a stable **bard-infra address** (a fabric name the coordinator resolves, independent of churny IPs / on-demand uptime).

**Key framing (holds up under review): Tailscale is *our* mesh; the coordinator is the *product's* mesh.** We don't need the coordinator to manage the fleet; the product needs it to reach customers' home nodes.

### The product MVP path (Basement) — build status per hop
Email QR → deep-link → redeem → enrolled → launch Squawk Box → LiveKit PTT to frogstation.

| Hop | Status |
|---|---|
| Invite create / redeem / membership / per-device token | **BUILT** (live-tested) |
| Router accepts per-device token | **BUILT** |
| Router→Registry `/agents/{id}` lookup with that token | **BROKEN — bug #63** (Registry gates fleet-only → 502) |
| Flutter deep-link handler, QR render, redeem UI, secure storage, JWT minting | **UNBUILT — all 5 platforms** |
| Squawk Box plugin **launch** (manifest `entry` execution) | **UNBUILT** |
| LiveKit integration + PTT UX | **UNBUILT** |
| frogstation as the Windows endpoint | **BLOCKED** on bootstrap |

**Honest completeness: backend ~done, client/voice ~0. The MVP is roughly 30% built — and it's the hard 30% (security/identity) that's done, with glue-code and one voice integration remaining.**

### Sharp architectural finding worth your call
**The Squawk Box plugin "kind" is ambiguous.** Its frozen manifest says `kind: "client"` with `["audio","network.lan"]`, but `entry.target` is a **Python** module path (`clients.squawk_box:create_client`). Is the plugin server-side (agent-hosted, client dials in) or client-side (Dart in the Flutter app)? This decides what "launch Squawk Box" even means and where LiveKit plugs in. **Recommendation: client-side** — manifest lives on the device, the Flutter app reads it and imports the Squawk Box Dart package directly (a normal app feature, not dynamic module loading), which then joins a LiveKit room. Freeze this in an ADR before P3.
